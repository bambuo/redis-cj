# RESP3 规范

## 版本历史
* 1.0, 2018年5月2日 — 初稿，征求社区反馈
* 1.1, 2018年11月5日 — 保留 CRLF 作为终止符 + 改进 "hello 回复" 章节
* 1.2, 2018年11月5日 — 根据开发者反馈进一步完善规范，未对协议进行实质性更改
* 1.3, 2019年3月11日 — 流式字符串和流式聚合类型

---

## 背景

Redis 协议在过去几年中表现良好，证明了如果设计得当，简单的人类可读协议不会成为客户端-服务器通信的瓶颈，且设计的简单性是创建健康的客户端库生态系统的主要优势。

然而，Redis 经验表明，在引入约六年后（自它取代初始 Redis 协议起），当前的 RESP 协议可以进行改进，尤其是在使客户端实现更简单和支持新功能方面。

同时，基本结构经过验证且可靠，旧设计中值得保留的特性：

* **RESP 是人类可读的** —— 易于观察和调试
* **性能不逊于二进制协议** — 在许多情况下更紧凑。协议的前缀长度用十进制数字表示，相比64位整数节省带宽，解析速度也不慢
* **设计简单** — 规范和实现简洁易懂

### RESP 的缺陷

* **语义能力不足**：客户端无法隐式了解适合的转换方式。例如 `LRANGE`、`SMEMBERS` 和 `HGETALL` 都返回数组（RESP v2 中的 multi bulk reply），但三种命令实际上分别返回数组、集合和映射。客户端需要记住每个命令才能进行正确的类型转换。
* **缺少关键数据类型**：浮点数和布尔值分别以字符串和整数返回；空值有双重表示（null bulk 和 null multi bulk），语义冗余。
* **无法返回二进制安全错误**：实现通用 API 时，必须检查并从错误字符串中移除可能的换行符。

### 新功能

* **通用推送模式**：相比 Redis 使用的 Pub/Sub，协议内置了对更通用的推送模式的支持。
* **带外数据**：如属性（Attribute）可以增强回复。
* **流式字符串**：支持预先不知道长度的字符串流式传输。

## 为什么现有序列化协议不够好？

理论上可以使用现有的序列化协议，如 MessagePack、BSON 等。但它们存在以下问题：

1. **非为请求-响应设计**：客户端和服务器需要额外约定上层协议
2. **复杂度更高**：客户端库需要依赖单独的序列化库，可能无法直接流式传输到 socket，导致效率低下
3. **缺少特定功能**：如预先未知长度的大字符串传输、数组的流式传输等

## 本规范的约定

为便于阅读，本规范在展示协议片段时使用两种格式：
* 转义字符串格式：如 `"foo\r\n"`
* 可读格式：`<CR><LF>` 表示 `\r\n`，后跟实际换行

嵌套聚合数据结构使用缩进表示：
```
*2<CR><LF>
    *2<CR><LF>
        :1<CR><LF>
        :2<CR><LF>
    #t<CR><LF>
```

## RESP3 概述

RESP3 是 RESP v2 的更新版本，设计用于处理客户端和服务器之间的请求-响应对话。

在 RESP3 中，客户端发送命令子集（仅数组类型），服务器可使用所有可用类型回复。

---

## RESP3 类型

### RESP2 已有的类型（在 RESP3 中继续支持）
* **数组 (Array)**：有序元素集合
* **二进制字符串 (Blob string)**：二进制安全字符串
* **简单字符串 (Simple string)**：空间高效的非二进制安全字符串
* **简单错误 (Simple error)**：空间高效的非二进制安全错误码和消息
* **数字 (Number)**：64 位有符号整数

### RESP3 新增的类型
* **空值 (Null)**：单值 null，替代 RESP v2 的 `*-1` 和 `$-1`
* **双精度浮点数 (Double)**：浮点数
* **布尔值 (Boolean)**：true 或 false
* **二进制错误 (Blob error)**：二进制安全的错误码和消息
* **逐字字符串 (Verbatim string)**：应直接显示给用户的字符串（无需转义）
* **映射 (Map)**：有序键值对集合
* **集合 (Set)**：无序元素集合
* **属性 (Attribute)**：类似 Map，客户端应读取但作为额外信息返回
* **推送 (Push)**：带外数据
* **大数 (Big number)**：Number 类型无法表示的大整数

## 简单类型详解

### 二进制字符串 (Blob string)

格式：`$<长度>\r\n<数据>\r\n`

与 RESP v2 完全相同。长度字段为无符号 64 位整数，0 为有效长度。

```
$11<CR><LF>
helloworld<CR><LF>
```

### 简单字符串 (Simple string)

格式：`+<字符串>\r\n`

简单字符串不能包含 CR 或 LF 字符。

```
+hello world<CR><LF>
```

### 简单错误 (Simple error)

与简单字符串相同，但首字节为 `-`：

```
-ERR this is the error description<CR><LF>
```

### 数字 (Number)

格式：`:<数字>\r\n`

```
:1234<CR><LF>
```

### 空值 (Null)

编码为 `_\r\n`：

```
_\r\n
```

### 双精度浮点数 (Double)

格式：`,<浮点数>\r\n`

特殊值：
```
,1.23\r\n
,inf\r\n       → 正无穷
,-inf\r\n      → 负无穷
```

### 布尔值 (Boolean)

```
#t\r\n   → true
#f\r\n   → false
```

### 二进制错误 (Blob error)

格式：`!<长度>\r\n<数据>\r\n`

```
!21<CR><LF>
SYNTAX invalid syntax<CR><LF>
```

### 逐字字符串 (Verbatim string)

格式：`=<长度>\r\n<3字节编码>:<数据>\r\n`

```
=15<CR><LF>
txt:Some string<CR><LF>
```

前 3 字节为格式信息（`txt` 或 `mkd`），第 4 字节为 `:`。

### 大数 (Big number)

格式：`(<大数>\r\n`

```
(3492890328409238509324850943850943825024385<CR><LF>
```

## 聚合数据类型

所有聚合类型的通用格式：

```
<聚合类型字符><元素数量><CR><LF> ... N个其他类型 ...
```

### 数组 (Array)

首字节为 `*`：

```
*3<CR><LF>
:1<CR><LF>
:2<CR><LF>
:3<CR><LF>
```

### 映射 (Map)

首字节为 `%`，字段-值对的数量：

```
%2<CR><LF>
+first<CR><LF>
:1<CR><LF>
+second<CR><LF>
:2<CR><LF>
```

### 集合 (Set)

首字节为 `~`，无序且元素唯一：

```
~5<CR><LF>
+orange<CR><LF>
+apple<CR><LF>
#t<CR><LF>
:100<CR><LF>
:999<CR><LF>
```

### 属性 (Attribute)

首字节为 `|`，与 Map 相同，但客户端应将其视为辅助数据而非回复的一部分。

```
|1<CR><LF>
    +key-popularity<CR><LF>
    %2<CR><LF>
        ...
*2<CR><LF>
    :2039123<CR><LF>
    :9543892<CR><LF>
```

### 推送 (Push)

首字节为 `>`。推送数据可出现在任何协议数据的顶层位置（但不会出现在聚合类型内部）。

```
>4<CR><LF>
+pubsub<CR><LF>
+message<CR><LF>
+somechannel<CR><LF>
+this is the message<CR><LF>
```

---

## 流式字符串 (Streamed Strings)

当预先不知道字符串大小时使用。

格式：
```
$?<CR><LF>
;<块长度><CR><LF><块数据><CR><LF>
...
;0<CR><LF>
```

以 `$?` 开始，后续使用 `;` 前缀的分块传输，以 `;0` 结束。

---

## 流式聚合数据类型 (Streamed Aggregated Data Types)

当聚合类型大小预先未知时，使用 `?` 替代长度，以 `.` 终止：

```
*?<CR><LF>
:1<CR><LF>
:2<CR><LF>
.<CR><LF>
```

同样支持 `~?`（流式集合）和 `%?`（流式映射，需要偶数个元素）。

## HELLO 命令与连接握手

语法：`HELLO <协议版本> [AUTH <用户名> <密码>]`

连接默认从 RESP2 模式开始。如果服务器不支持请求的协议版本，返回 `-NOPROTO` 错误。

HELLO 成功回复为 Map，包含以下必填字段：
* **server**: "redis"（或其他软件名称）
* **version**: 服务器版本
* **proto**: 支持的最高 RESP 协议版本

Redis 特有字段：
* **id**: 客户端连接 ID
* **mode**: "standalone", "sentinel" 或 "cluster"
* **role**: "master" 或 "replica"
* **modules**: 已加载模块的字符串数组

## FAQ

**问：为什么 RESP3 没有将换行符改为单个字符？**
答：因为客户端需要根据当前协议版本（RESP v2 或 v3）以不同方式发送命令和解析回复。即使在 RESP3 模式下，连接初始时也处于 RESP v2 模式，需要发送 HELLO 命令才能切换。为保持向后兼容，保留 CRLF 作为终止符是最实际的选择。
